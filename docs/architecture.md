# Architecture Overview

## High-Level Design

Civyx is composed of three protocol layers, nine core contracts, eight supporting contracts, and three ZK circuits — all deployed on Polkadot Asset Hub Testnet (chainId: 420420417).

The layers are not independent modules. Each one depends on the one below it. Reputation requires identity. CivUSD requires reputation. This dependency chain is intentional — it means you cannot game the top layer without first satisfying the bottom one.

```
┌─────────────────────────────────────────────┐
│              DeFi Layer                      │
│         CivUSD Stablecoin                    │
│   Reputation-tiered collateral ratios        │
└─────────────────────────┬───────────────────┘
                          │ reads reputation score
┌─────────────────────────▼───────────────────┐
│            Reputation Layer                  │
│     ReputationRegistry + TrustOracle         │
│   Task system + external verification        │
└─────────────────────────┬───────────────────┘
                          │ requires verified identity
┌─────────────────────────▼───────────────────┐
│             Identity Layer                   │
│   IdentityRegistry + ZK circuits             │
│   Wallet linking + nullifier tracking        │
└─────────────────────────────────────────────┘
```

## The Three Layers

### Identity Layer

The foundation of the entire protocol. Before a user can earn reputation or mint CivUSD, they must register an identity.

An identity in Civyx is a cryptographic commitment — a hash of a secret that only the user knows — stored on-chain alongside a PAS stake. The user proves ownership of that commitment using a ZK proof without ever revealing the secret itself. Additional wallets can be linked to the same identity using the wallet link ZK circuit, so all linked wallets share the same reputation score and borrowing power.

**Key contracts:** `IdentityRegistry`, `WalletLinkVerifier`, `IdentityVerifierRouter`, `IdentityBroadcaster`

---

### Reputation Layer

Built directly on top of identity. Every action that earns reputation requires a verified identity underneath it.

Reputation is a score from 0 to 1000, accumulated through task completion. Tasks are defined by organizers — entities registered in the `OrganizerRegistry` — and dispensed through the `TaskRewardDispenser`. Verification happens two ways: internal tasks verify actions within the Civyx ecosystem itself; external tasks verify actions on entirely separate contracts using `staticcall` — no integration required from the target protocol.

**Key contracts:** `ReputationRegistry`, `OrganizerRegistry`, `TrustOracle`, `TaskRewardDispenser`, `ExternalTaskVerifier`

---

### DeFi Layer

The layer that gives reputation financial weight. CivUSD is a collateral-backed stablecoin where the collateral ratio scales with the user's reputation score. The `TrustOracle` is queried at mint time and the ratio is calculated on-chain. No manual process. No human review.

**Key contracts:** `CivUSD`

## ZK Circuit Layer

Three Noir circuits sit underneath the identity layer, compiled to UltraHonk proofs and verified on-chain via Solidity verifier contracts generated by Barretenberg.

| Circuit | Purpose | Public Inputs |
|---|---|---|
| `identity` | Proves knowledge of secret behind a commitment | commitment |
| `wallet_link` | Links a new wallet to an existing identity | commitment, nullifier, wallet_address |
| `nullifier` | General-purpose one-time proof for any action | commitment, nullifier, action_id |

All circuits use BN254 curve arithmetic, compiled with Noir 1.0.0-beta.18 and Barretenberg 3.0.0-nightly.20260102.

## Design Principles

**Composability over isolation.** Every piece of Civyx state is readable by external contracts. Reputation scores, identity status, nullifier flags — all queryable via standard view functions. Any protocol on Polkadot can integrate Civyx as a trust source without permission.

**Privacy by default.** The secret behind an identity never touches the chain. Wallet linking never reveals which wallets belong to the same identity. ZK proofs are generated entirely in the user's browser.

**No trusted intermediaries.** There is no oracle operator, no admin key that can alter scores, no multisig that controls minting. Access control roles exist for operational safety but the core verification logic is immutable circuit math.

**Upgradeable where it matters, fixed where it counts.** Pausability and role-based access control are built in via OpenZeppelin. The ZK verification logic itself is not upgradeable — a verifier contract is a deterministic function of the circuit it was compiled from.
