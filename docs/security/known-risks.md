# Known Risks and Limitations

This page is an honest accounting of every known risk in the current Civyx implementation. Civyx is a testnet deployment. Understanding these limitations is important before building on or integrating the protocol.

---

## No Formal Audit

Civyx has not undergone a formal security audit. The contracts use audited OpenZeppelin components throughout and were written with security best practices in mind, but independent expert review has not been performed.

**Impact:** Unknown vulnerabilities may exist.
**Mitigation:** Testnet deployment only — no real capital at risk.
**Planned fix:** Formal audit by a reputable firm (Trail of Bits, Spearbit, or Zellic) before any mainnet deployment. The audit scope will include all 17 contracts, ZK circuit implementations, and XCM message handling.

---

## Admin Key Centralization

The `DEFAULT_ADMIN_ROLE` across all 17 contracts is held by a single deployer EOA. A compromised admin key could inflate reputation scores, drain protocol fees, update collateral ratios to unsafe values, or pause all contracts simultaneously.

**Impact:** High — complete protocol compromise possible.
**Mitigation:** Testnet deployment only. No real capital at risk.
**Planned fix:** Transfer `DEFAULT_ADMIN_ROLE` to a Gnosis Safe multisig (3-of-5 minimum) before any mainnet deployment.

---

## Secret Loss Is Permanent

If a user loses their secret and has no backup, their identity becomes permanently frozen. No new wallets can be linked. No new ZK proofs can be generated. There is no recovery mechanism.

**Impact:** User loses access to identity extension capabilities permanently.
**Mitigation:** Frontend generates a downloadable backup file at registration and warns the user before proceeding.
**Planned fix:** None — any recovery mechanism would require a trusted party, which defeats the purpose of the trustless identity system. User education is the only mitigation.

---

## Simplified Price Oracle

CivUSD's liquidation logic depends on a PAS/USD price feed. The current testnet implementation uses a simplified oracle that is not manipulation-resistant.

**Impact:** Liquidations could fire incorrectly (price too low) or fail to fire when needed (price too high).
**Mitigation:** Testnet deployment only.
**Planned fix:** Replace with a robust, decentralized price feed before mainnet — either a Polkadot-native oracle or a Chainlink integration via XCM.

---

## ZK Circuit Trust Assumptions

The ZK verifier contracts are generated by Barretenberg from compiled Noir circuits. Security depends on:

- Correctness of the Noir compiler
- Correctness of Barretenberg's proof system implementation
- Absence of bugs in the circuit logic itself

None of these have been formally verified for this specific deployment.

**Impact:** Circuit bugs could theoretically allow proofs for false statements.
**Mitigation:** Circuits are simple (2–4 lines of Noir) and peer-reviewed internally.
**Planned fix:** ZK circuit formal verification as part of the full audit.

---

## Proof System Version Lock

The deployed verifier contracts are generated from Barretenberg 3.0.0-nightly.20260102. If a vulnerability is discovered in this version, the verifier contracts cannot be upgraded in place.

**Impact:** Requires deploying new verifier contracts and updating the `IdentityVerifierRouter` — a coordinated migration.
**Mitigation:** The `IdentityVerifierRouter` exists specifically to enable this migration without redeploying `IdentityRegistry`.
**Planned fix:** Monitor Barretenberg security advisories. The router upgrade path is designed and ready.

---

## Reputation Score Immutability

Once reputation points are awarded, they cannot be revoked. If a task contract has an exploitable verification bug, points awarded through the exploit cannot be removed.

**Impact:** Score inflation from a compromised task contract.
**Mitigation:** Trust level caps bound damage — a trust level 1 exploit awards at most 5 points per completion. Task contracts are admin-authorized before being activated.
**Planned fix:** Consider an emergency score correction function gated behind a high-threshold multisig for post-audit deployment.

---

## Single Position Per Wallet

Each wallet can hold only one active CivUSD position. Users cannot hedge exposure across multiple positions at different collateral ratios.

**Impact:** Reduced capital flexibility for advanced users.
**Mitigation:** Documented limitation — users can use multiple linked wallets for multiple positions.
**Planned fix:** Multi-position support per wallet on the roadmap.

---

## Tier 5 Opens Below Liquidation Threshold

Users with elite reputation (tier 5) open CivUSD positions at a 110% collateral ratio, which is already below the 120% liquidation threshold. These positions require active collateral management from day one.

**Impact:** Tier 5 positions can be liquidated immediately if PAS price drops even slightly.
**Mitigation:** Documented behavior — tier 5 users are expected to understand and actively manage this risk.
**Planned fix:** Consider raising the minimum opening ratio to 115% for tier 5, or implementing a grace period before liquidation eligibility.

---

## XCM No Destination Receivers Deployed

No Civyx receiver contracts are deployed on any destination parachain. XCM broadcasting is implemented and functional on Asset Hub, but the other end of the system does not exist yet.

**Impact:** Cross-chain reputation portability is not usable in the current testnet state.
**Mitigation:** Clearly documented as in-progress.
**Planned fix:** Receiver deployment on Westend and Rococo as the first XCM milestone.
